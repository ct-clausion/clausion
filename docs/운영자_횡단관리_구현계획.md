# 운영자 횡단 관리 구현 계획

**핵심 원칙**: 운영자 = 강사가 못하는 것을 하는 사람 (과정 횡단 비교/분석/조율)
**날짜**: 2026-04-12
**Branch**: yc

---

## 1. 운영자 재정의

```
강사: 자기 과정 안에서 학생을 가르치고 관리
운영자: 여러 과정/강사를 횡단해서 비교, 분석, 조율, 행정

강사가 볼 수 있는 것       운영자만 볼 수 있는 것
─────────────────────     ─────────────────────────
내 과정의 학생 목록          과정 A vs 과정 B 성과 비교
내 학생의 Twin 데이터        강사 A vs 강사 B 효과성
내 상담 이력                강사별 상담 부하 균형
내 과정의 커리큘럼           전체 과정 편성/승인
                           강사에게 개입 지시 발송
                           출결/수료 행정
                           감사 로그
```

## 2. 현재 페이지 분류

### A. 유지 (운영자 고유 가치 있음)

| # | 페이지 | 현재 상태 | 보강 내용 |
|---|--------|----------|----------|
| 1 | 대시보드 | KPI + Course Twin 바 | Course Twin 비교 차트, 강사 부하 요약 추가 |
| 2 | 출결 관리 | 과정/세션 선택 UI | 세션 생성, 벌크 체크, 출결 통계 차트 완성 |
| 3 | 공지사항 | CRUD 동작함 | 읽음 현황 상세, 대상 과정 선택 |
| 4 | AI 리포트 | 텍스트 표시 | 강사 효과성 비교 차트 추가 |
| 5 | AI 시뮬레이션 | 하드코딩 결과 | 강사 재배정 시뮬레이션 추가 |
| 6 | 감사 로그 | 테이블 + 페이징 | 필터(행위/날짜) 추가 |

### B. 재설계 (강사와 중복 → 횡단 관리 도구로 변경)

| # | 현재 | 문제 | 재설계 방향 |
|---|------|------|-----------|
| 7 | 과정 관리 (목록) | 강사가 자기 과정 이미 관리 | → **과정 편성 워크플로우**: 승인/반려 + 강사 배정 + 정원 관리. 목록이 아닌 "결재함" 컨셉 |
| 8 | 강사 관리 (카드 목록) | 목록만 보는 건 가치 없음 | → **강사 효과성 대시보드**: 강사 간 성과 비교, 업무 부하 균형, 학생 재배정 |
| 9 | 수강생 관리 (테이블) | 강사의 "학생 모니터링"과 동일 | → **제거** (수강생 정보는 강사 효과성 대시보드에서 집계로만 활용) |
| 10 | 이탈 위험군 (목록) | 강사의 Twin 대시보드와 동일 | → **강사 개입 지시 센터**: 운영자가 강사에게 "이 학생 주의 바람" 알림을 보내는 도구 |

### C. 신규 구현 (강사가 못하는 것)

| # | 기능 | 설명 | 우선순위 |
|---|------|------|---------|
| 11 | 강사 효과성 비교 차트 | 강사별 학생 Twin 개선율, 상담 후 성과 변화 비교 | 1순위 |
| 12 | 업무 부하 균형 뷰 | 강사별 학생 수/상담 수/과정 수 한눈에 + 재배정 제안 | 1순위 |
| 13 | 강사 개입 지시 | "이 학생 주의 바람" → 강사에게 알림 전송 + 추적 | 1순위 |
| 14 | 출결 세션 생성 + 통계 차트 | 세션 CRUD 완성 + Recharts 출결률 차트 | 2순위 |
| 15 | 과정 승인 워크플로우 상세 | 승인/반려 사유, 강사 배정 모달, 상태 전환 | 2순위 |
| 16 | AI 리포트 강사 비교 섹션 | "강사 A의 학생 숙련도 평균 72, B는 58" 시각화 | 2순위 |

## 3. 구현 계획

### Phase 1: 재설계 + 핵심 횡단 기능 (1순위)

#### 3.1 강사 효과성 대시보드 (강사 관리 → 재설계)

현재 `InstructorManagement.tsx` (카드 목록) → 완전 재작성

**새 구성**:
```
┌─────────────────────────────────────────────────┐
│  강사 효과성 분석                                 │
│                                                   │
│  ┌─── 업무 부하 균형 ─────────────────────────┐   │
│  │ 강사A ████████░░ 학생 25 / 상담 8 / 과정 3 │   │
│  │ 강사B ████░░░░░░ 학생 12 / 상담 3 / 과정 1 │   │
│  │ 강사C ██████████ 학생 30 / 상담 12 / 과정 3│   │
│  │         [부하 불균형 경고: C 과부하]            │   │
│  └───────────────────────────────────────────┘   │
│                                                   │
│  ┌─── 학생 성과 비교 (Twin 개선율) ─────────┐      │
│  │ [Recharts BarChart]                      │      │
│  │ 강사A: 숙련도 +12, 동기 +8              │      │
│  │ 강사B: 숙련도 +5, 동기 +15              │      │
│  │ 강사C: 숙련도 -3, 동기 -5  ← 주의      │      │
│  └──────────────────────────────────────────┘      │
│                                                   │
│  ┌─── 상담 효과 비교 ──────────────────────┐      │
│  │ 상담 후 Twin 점수 변화 (강사별)         │      │
│  │ [Recharts GroupedBarChart]               │      │
│  └──────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────┘
```

**백엔드 API**:
```
GET /api/operator/instructors/effectiveness
  → 강사별 { id, name, studentCount, consultationCount, courseCount,
               avgMasteryImprovement, avgMotivationImprovement,
               consultationEffectiveness }

GET /api/operator/instructors/workload
  → 강사별 { id, name, studentCount, consultationCount, courseCount,
               workloadScore, isOverloaded }
```

#### 3.2 강사 개입 지시 센터 (이탈 위험군 → 재설계)

현재 `AtRiskStudents.tsx` (목록 + 개입 승인) → 재설계

**핵심 변경**: 운영자가 직접 학생을 보는 게 아니라, **강사에게 지시를 보내는** 도구

**새 구성**:
```
┌─────────────────────────────────────────────────┐
│  강사 개입 지시 센터                               │
│                                                   │
│  AI가 감지한 이슈 (강사별 그룹)                     │
│                                                   │
│  ┌─ 강사A (Python 과정) ─────────────────────┐  │
│  │ 위험 학생 3명: 김OO, 이OO, 박OO              │  │
│  │ AI 제안: "3명 동시 보충 세션 권장"             │  │
│  │ [강사에게 알림 보내기] [상세]                 │  │
│  └────────────────────────────────────────────┘  │
│                                                   │
│  ┌─ 강사B (React 과정) ─────────────────────┐  │
│  │ 위험 학생 1명: 최OO                         │  │
│  │ AI 제안: "1:1 상담 배정 권장"                │  │
│  │ [강사에게 알림 보내기] [상세]                 │  │
│  └────────────────────────────────────────────┘  │
│                                                   │
│  ┌─ 지시 이력 ────────────────────────────────┐  │
│  │ 2026-04-10 강사A에게 보충 세션 지시 → 처리됨 │  │
│  │ 2026-04-08 강사B에게 상담 배정 지시 → 대기중 │  │
│  └────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

**백엔드 API**:
```
GET  /api/operator/intervention-center
  → 강사별로 그룹핑된 위험 학생 + AI 제안

POST /api/operator/intervention-directive
  → { instructorId, studentIds[], directiveType, message }
  → 강사에게 Notification 발송 + 감사 로그

GET  /api/operator/intervention-directives
  → 지시 이력 (발송/처리/대기)
```

```

  ├── Frontend: StudentManagement.tsx 제거, App.tsx 라우트 제거

  └── CourseManagement: 강사 배정 모달 추가

  ├── Dashboard: Recharts Course Twin 비교 차트
  ├── Dashboard: 강사 부하 요약 위젯
  ├── Attendance: 세션 생성 + 벌크 저장 완성

Step 5: AI 고도화
  ├── AI 리포트: 강사 비교 섹션 추가

```

## 6. 서브 이슈 (매칭이 부자연스러운 것)

현재 구현에서 강사 횡단 관점과 맞지 않아 서브로 남기는 항목:

| # | 항목 | 왜 서브인가 | 나중에 필요할 때 |
|---|------|-----------|-------------|
| S1 | 수료/성적 관리 | 횡단 비교보다는 행정 업무. 강사가 성적 입력 → 운영자가 수료 판정은 가능하지만 현재 성적 입력 자체가 없음 | 성적 입력 기능이 강사 쪽에 먼저 필요 |
| S2 | 만족도 조사 | 운영자 고유 업무이긴 하지만 설문 시스템 자체가 없음 | 설문 도메인을 먼저 설계해야 |
| S3 | 과정 시간표/캘린더 | 시간표 충돌 관리는 운영자 업무지만, 현재 과정에 시간 데이터가 free-text | 구조화된 시간 데이터 먼저 필요 |
| S4 | 엑셀/PDF 내보내기 | 횡단 비교 자체와 무관. 결과물 출력 기능 | 어떤 데이터를 내보낼지 확정 후 |
| S5 | CourseTwinCalculator 야간 배치 | 현재 on-the-fly 집계로 동작함. 스케줄러는 최적화 | 데이터 규모가 커지면 |

## 7. 핵심 검증 기준

구현 후 이 질문에 답할 수 있어야 한다:

> "이 화면에서 강사가 할 수 없는 것은 무엇인가?"

각 페이지별 답변:
- **대시보드**: "여러 과정의 건강도를 한눈에 비교할 수 있다" (강사는 자기 과정만 봄)
- **강사 분석**: "강사 간 성과를 비교하고 업무 부하를 조정할 수 있다"
- **개입 지시 센터**: "강사에게 특정 학생에 대한 주의 지시를 보낼 수 있다"
- **과정 편성**: "과정 승인/반려하고 강사를 배정할 수 있다"
- **출결 관리**: "전체 과정의 출결 행정을 처리할 수 있다"
- **AI 리포트**: "강사 간 효과성을 비교 분석받을 수 있다"
- **AI 시뮬레이션**: "강사 재배정/보충 세션 추가의 예상 효과를 볼 수 있다"
