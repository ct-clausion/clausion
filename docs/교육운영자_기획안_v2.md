# Clausion - 교육 운영자(Education Operator) 기획안 v2

**문서 버전**: v2.0  
**작성일**: 2026-04-12  
**기술 스택**: React 19 + TypeScript / Spring Boot 3.3.5 + Java 17 / PostgreSQL 16  
**v1 대비 변경**: Claude 비평 + OpenAI Codex 독립 분석 피드백 전면 반영

---

## 1. Executive Summary

v1 기획안은 "관리자 CRUD + LLM 요약"에 불과했다. v2는 **AI Operations Copilot** — 교육 운영자가 매일 사용하는 의사결정 도구로 재설계한다.

**히어로 피처**: Course Twin (과정 수준 디지털 트윈)  
**차별화**: What-if 시뮬레이션, AI 자동 개입 워크플로우, 출결 기반 이탈 예측

## 3. 교육 운영자 역할 정의

### 3.1 핵심 정의

> 교육 운영자는 전체 교육 과정의 건강 상태를 모니터링하고, AI 인사이트에 기반하여 선제적 개입을 수행하며, 교강사-수강생 간 교육 품질을 최적화하는 운영 총괄자이다.

### 3.2 기존 역할과의 관계

```
교육 운영자 (OPERATOR)
├── 전체 과정 횡단 관리 (Course Twin 모니터링)
├── 교강사 업무 배분 최적화
├── 이탈 위험군 AI 자동 개입
├── 출결/수료/공지 관리
└── 운영 의사결정 AI 지원

교강사 (INSTRUCTOR)          수강생 (STUDENT)
├── 자기 과목 내 교육         ├── 학습 활동
├── 학생 상담/평가            ├── 상담 요청
└── 문제 출제                └── 스터디그룹 참여
```

## 4. "운영자의 하루" 사용자 여정

```
09:00  로그인 → 대시보드에서 야간 AI 분석 결과 확인
       "Java과정 3주차 위험도 급등, 수강생 4명 이탈 위험"
       
09:15  이탈 위험군 페이지 → AI가 제안한 자동 개입 워크플로우 검토
       "김OO, 이OO에게 보충 상담 자동 배정 → [승인] 클릭"
       
09:30  출결 현황 확인 → 오늘 결석자 3명, AI가 패턴 분석
       "박OO 3일 연속 지각 → 이탈 확률 78% → 교강사에게 알림"
       
10:00  과정 관리 → 교강사가 요청한 신규 과정 승인
       교강사 업무량 확인 후 배정 최적화
       
11:00  AI 시뮬레이션 → "만약 3주차에 보충 세션을 추가하면?"
       → "예상 수료율 72% → 85% (+13%p)" → 보충 세션 승인
       
14:00  공지사항 작성 → 전체 공지 + 특정 과정 공지
       읽음 현황 실시간 추적
       
16:00  주간 AI 리포트 확인 → 과정별 스코어카드
       교강사 효과성 비교, 개입 효과 추적
```

## 5. 기능 설계

### Phase 1: 인프라 + 핵심 운영 (MVP)

#### 5.1 OPERATOR 역할 인프라

**보안 강화 (Codex 지적 반영)**:
- 등록 시 OPERATOR 역할 선택 차단 (AuthController에서 role 검증)
- 운영자 계정은 시드 데이터 또는 기존 운영자가 초대하는 방식
- `/api/operator/**` 엔드포인트에 `hasRole("OPERATOR")` 적용
- 기존 공유 엔드포인트에 `@PreAuthorize` 메서드 레벨 보안 추가

**변경 파일 목록**:
- `User.java`: Role enum에 OPERATOR 추가
- `AuthController.java`: register 시 OPERATOR 역할 차단
- `AuthDtos.java`: RegisterRequest role validation
- `SecurityConfig.java`: `/api/operator/**` 규칙 추가
- `types/index.ts`: User.role에 'OPERATOR' 추가
- `App.tsx`: operator 라우트 추가
- `AppShell.tsx`: role prop 'operator' 추가
- `Sidebar.tsx`: operatorNav 추가
- `Register.tsx`: OPERATOR 선택 불가 유지
- `Login.tsx`: 로그인 후 operator 리다이렉트

#### 5.2 운영자 대시보드 & Course Twin

**Course Twin (히어로 피처)**:
기존 StudentTwin의 8차원 점수를 과정 수준으로 집계하여 "과정의 건강도"를 표현

```
Course Twin 지표:
├── avg_mastery: 과정 내 전체 수강생 평균 숙련도
├── avg_execution: 과정 내 전체 수강생 평균 실행력
├── avg_motivation: 과정 내 전체 수강생 평균 동기
├── retention_risk_ratio: 이탈 위험(>0.7) 수강생 비율
├── consultation_backlog: 미처리 상담 건수
├── attendance_rate: 최근 7일 출석률
├── completion_projection: AI 예측 수료율
└── health_score: 종합 건강 점수 (0-100)
```

**대시보드 위젯**:
- 과정 현황 카드 (진행중/예정/완료)
- Course Twin 히트맵 (과정별 건강도 시각화)
- 이탈 위험 실시간 알림
- 출결 요약 (오늘 출석률, 결석자)
- 주간 KPI 추세

#### 5.3 출결 관리 (First-class 도메인)

**데이터 모델**:
```sql
-- 수업 세션 (과정의 각 수업 단위)
CREATE TABLE course_sessions (
    id BIGSERIAL PRIMARY KEY,
    course_id BIGINT REFERENCES courses(id) NOT NULL,
    session_date DATE NOT NULL,
    session_number INTEGER NOT NULL,
    title VARCHAR(200),
    status VARCHAR(20) DEFAULT 'SCHEDULED',
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(course_id, session_date, session_number)
);

-- 출결 기록
CREATE TABLE attendance_records (
    id BIGSERIAL PRIMARY KEY,
    session_id BIGINT REFERENCES course_sessions(id) NOT NULL,
    student_id BIGINT REFERENCES users(id) NOT NULL,
    status VARCHAR(20) NOT NULL,  -- PRESENT, ABSENT, LATE, EXCUSED
    check_in_time TIMESTAMP,
    recorded_by BIGINT REFERENCES users(id),
    note TEXT,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(session_id, student_id)
);
```

**기능**:
- 일별 출석 체크 (PRESENT/ABSENT/LATE/EXCUSED)
- 벌크 출석 처리 (전체 출석, 선택 결석)
- 출석률 자동 집계 (과정별, 수강생별)
- AI 출결 패턴 분석: "3일 연속 지각 → 이탈 확률 X%"
- 출결 데이터 → Twin 추론 입력으로 연동

#### 5.4 과정 운영 관리

**과정 승인 워크플로우**:
- Course 엔티티에 `approval_status` 추가 (PENDING → APPROVED / REJECTED)
- 교강사가 과정 생성 시 PENDING 상태로 시작
- 운영자가 검토 후 승인/반려 + 사유 기록
- 교강사 배정: `course_instructor_assignments` JOIN 테이블

**교강사 업무량 대시보드**:
- 교강사별 담당 과정 수, 수강생 수, 상담 건수
- AI 업무 균형 분석: 최적 배분 제안

#### 5.5 인원 관리 + 이탈 위험군

**이탈 위험군 관리 (AI 자동 개입)**:
- Twin overall_risk_score > 0.7인 수강생 자동 목록화
- AI가 개입 방법 제안 (상담 배정, 보충 자료, 메시지 발송)
- 운영자 승인 → 자동 실행
- 개입 효과 추적 (개입 전/후 Twin 변화)

**개입 기록**:
```sql
CREATE TABLE intervention_logs (
    id BIGSERIAL PRIMARY KEY,
    student_id BIGINT REFERENCES users(id) NOT NULL,
    operator_id BIGINT REFERENCES users(id) NOT NULL,
    course_id BIGINT REFERENCES courses(id),

    description TEXT NOT NULL,

    twin_score_after JSONB,

    status VARCHAR(20) DEFAULT 'PENDING', -- PENDING, IN_PROGRESS, COMPLETED, INEFFECTIVE
    created_at TIMESTAMP DEFAULT NOW(),
    completed_at TIMESTAMP
);
```

### Phase 2: AI Operations Copilot

#### 5.8 AI What-if 시뮬레이션

**핵심 기능**: 운영자가 가상의 개입 시나리오를 실행하고 예상 결과를 확인

시뮬레이션 시나리오:
1. "이 수강생에게 상담을 배정하면 Twin 점수가 어떻게 변할까?"
2. "3주차에 보충 세션을 추가하면 과정 수료율이 어떻게 변할까?"
3. "교강사 A의 상담 부하를 줄이면 전체 상담 품질이 어떻게 변할까?"

**구현 방식**:
- 기존 `TwinInferenceEngine`의 규칙 기반 점수 계산 로직을 추출
- 가상 데이터를 주입하여 점수 변화를 예측
- LLM이 시뮬레이션 결과를 해석하고 권장사항 생성

**API**:
```
POST /api/operator/ai/simulate
Body: {
  "scenario_type": "ADD_CONSULTATION" | "ADD_SESSION" | "REBALANCE_INSTRUCTOR",
  "target_student_id": 123,
  "target_course_id": 456,
  "parameters": { ... }
}
Response: {
  "current_scores": { ... },
  "projected_scores": { ... },
  "confidence": 0.82,
  "ai_interpretation": "상담 배정 시 동기 점수 +12, 이탈 위험 -15% 예상",
  "recommendation": "즉시 상담 배정을 권장합니다"
}
```

#### 5.9 AI 자동 개입 워크플로우

**프로세스**:
```
야간 Twin 추론 (02:00 UTC)
    ↓
이상 징후 감지 (overall_risk > 0.7 또는 급격한 변화)
    ↓
AI가 개입 방법 생성 (상담 배정, 메시지, 보충 자료)
    ↓
운영자 대시보드에 "승인 대기" 카드로 표시
    ↓
운영자 [승인] → 자동 실행 (상담 생성, 알림 발송)
    ↓
7일 후 Twin 재추론 → 효과 측정
```

#### 5.10 AI 주간 운영 리포트

- 기존 `openAiRestTemplate` + `AiJobService` 패턴 활용
- 입력: Course Twin 집계 + 출결 데이터 + 상담 이력 + 개입 효과
- 출력: 구조화된 JSON 리포트 (요약, 이상징후, 교강사 효과성, 권장 조치)
- 비동기 생성 (기존 @Async 패턴)

## 8. AI 차별화 상세

### 8.1 Course Twin (코호트 디지털 트윈)
- 기존 `TwinInferenceEngine` + `TwinDataCollector` 집계
- 과정 내 전체 StudentTwin 평균/분포/추세 계산
- 야간 배치(02:00 UTC)에서 `CourseTwinCalculator`가 스냅샷 생성
- 건강 점수 산출: `health = 0.3*mastery + 0.2*execution + 0.2*motivation + 0.15*(1-risk_ratio) + 0.15*attendance`

### 8.2 What-if 시뮬레이션
- `TwinInferenceEngine`의 규칙 기반 점수 계산을 분리하여 시뮬레이션 모드로 실행
- 가상 개입 데이터 주입 → 점수 변화 예측 → LLM이 해석
- 시뮬레이션 결과는 DB에 저장하지 않음 (읽기 전용)

### 8.3 AI 자동 개입
- Twin 이상 징후 감지 → AI가 개입 방법 생성 → 운영자 승인 → 자동 실행
- 개입 유형: 상담 자동 배정, 교강사 알림, 보충 자료 추천
- 효과 측정: 개입 전/후 Twin 점수 비교

### 8.4 출결 기반 이탈 예측
- 출결 패턴 → Twin retention_risk_score에 가중 반영
- "3일 연속 결석 → 이탈 확률 X%" 자동 계산
- Twin 추론 입력에 출결 데이터 추가
